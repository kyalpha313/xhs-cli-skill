# Safety Rules

这份文档定义 `xhs-cli-skill` 的全局安全规则。

## 一、总原则

- 只包装稳定 CLI 能力
- 默认优先只读路径
- 写操作必须确认
- 遇到边界外请求时先说明限制，不做静默尝试

## 二、写操作规则

以下命令都属于写操作：

- `like`
- `like --undo`
- `favorite`
- `unfavorite`
- `comment`
- `reply`
- `delete-comment`
- `follow`
- `unfollow`

执行前必须做到：

1. 目标明确
2. 参数明确
3. 用户确认
4. 登录状态有效

如果其中任一条件不满足，应先补齐信息或终止执行。

## 三、登录规则

- 登录问题先走 `xhs auth doctor`
- 未登录时，不直接重试业务命令
- 故障恢复优先短路径：`login`、`auth import --file`、`auth import-fields --interactive`
- 非必要时不向用户灌输底层 cookie 细节

## 四、频率与范围规则

- 单次只处理用户明确指定的目标
- 不主动建议批量高频互动
- 不自动扩展到“顺手多做几条”
- 不把单条请求升级成批处理任务

## 五、能力边界规则

- 不默认调用隐藏命令
- 不调用已知失败命令
- 当用户触发边界外能力时，应明确说明当前不承诺
- 若存在稳定替代路线，应优先提供替代路线

## 六、输出规则

- 默认输出摘要
- 保留关键字段和下一步建议
- 只有在用户需要时才展示更长原始结果
- 使用替代路线时必须明确标记

## 七、错误处理规则

发现以下情况时应立即切换到解释模式，而不是继续重试：

- 未登录或会话失效
- 参数缺失
- 已知失败能力
- 平台风控或校验异常
- 临时网络错误反复出现

## 八、禁止行为

- 默认调用 `post`
- 默认调用 `delete`
- 默认调用 `favorites`
- 默认调用 `likes`
- 默认调用 `notifications`
- 默认调用 `user` / `user-posts`
- 默认进行批量自动互动

## 九、对用户的表达方式

安全限制不应只说“不能做”，而应优先使用下面的表达方式：

- 先解释当前边界
- 再给出稳定替代方案
- 再说明下一步建议

示例：

- 用户要看收藏列表：建议改走 `board`
- 用户要看新通知：建议先用 `unread`
- 用户要删除内容：说明当前 skill MVP 不承诺该能力
